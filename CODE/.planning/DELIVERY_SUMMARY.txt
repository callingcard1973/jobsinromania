================================================================================
RISK MITIGATION
================================================================================

Risk: PG connection pool exhaustion
Severity: LOW
Mitigation: Pool min=1, max=5. Monitor with psycopg2.pool logging.

Risk: Data corruption during migration
Severity: MEDIUM
Mitigation: Idempotent migration (ON CONFLICT), validate row counts + emails.

Risk: Duplicate sends (parallel run)
Severity: MEDIUM
Mitigation: Offset sender times by 30min, compare email sets daily.

Risk: Brevo quota exceeded
Severity: LOW
Mitigation: Each sender has independent quota. Parallel splits load.

Risk: Connection pool exhaustion
Severity: LOW
Mitigation: Pool logging, query active connections post-send.

Risk: Rollback takes too long
Severity: LOW
Mitigation: SQLite backup on both devices. Restore < 1 minute.

================================================================================
ROLLBACK PLAN (< 1 HOUR)
================================================================================

If PG fails during parallel run or cutover:

Option 1: Revert to SQLite
  1. Disable PG sender in orchestrator config
  2. Restore SQLite: cp tudor_backup_*.db tudor.db
  3. Revert run-tudor.sh to tudor_sender.py
  4. Restart orchestrator
  Timeline: ~10 minutes

Option 2: Purge and Retry (if corruption detected)
  1. DELETE FROM anofm_contacts (with approval)
  2. Re-run migration script
  3. Resume parallel run
  Timeline: ~20 minutes

Full Recovery: PostgreSQL WAL (point-in-time)
  Enabled by default. Can restore to any point in last 24h.
