## LOCK CONTENTION FIX
================================================================================

PROBLEM:
- 35+ concurrent DELETE queries competing
- 1109 RowExclusiveLocks
- Queries running 2-3 hours each
- 5 deadlocks recorded

ROOT CAUSE:
- dedup_manager.py had no instance lock
- Cron spawned new process every 5 min while previous still running

FIX:
1. Added fcntl.LOCK_EX lock file to dedup_manager.py
2. Killed 32 competing DELETE queries
3. Reduced batch size 100K -> 50K for single-thread
4. Created parallel version with 6 workers

RESULT:
- RowExclusiveLocks: 1109 -> 3
- No more competing queries
- Dedup completed in 20 minutes

## USEFUL COMMANDS
================================================================================

# Database health
python3 /opt/ACTIVE/DB/db_health.py

# Slow queries
sudo -u postgres psql -d interjob_master -c "
SELECT left(query, 60), calls, round(total_exec_time::numeric/1000,1) as total_sec
FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10;"

# Check duplicates
sudo -u postgres psql -d interjob_master -c "
SELECT COUNT(*) FROM (
  SELECT lower(email), COUNT(*) FROM companies
  WHERE email IS NOT NULL GROUP BY lower(email) HAVING COUNT(*) > 1
) x;"

# Check locks
sudo -u postgres psql -c "
SELECT locktype, mode, COUNT(*) FROM pg_locks GROUP BY locktype, mode ORDER BY count DESC;"

# Active queries
sudo -u postgres psql -c "
SELECT pid, now() - query_start as duration, left(query, 50)
FROM pg_stat_activity WHERE state = 'active' ORDER BY query_start;"

# Database sizes
sudo -u postgres psql -c "
SELECT datname, pg_size_pretty(pg_database_size(datname))
FROM pg_database ORDER BY pg_database_size(datname) DESC;"

# Table sizes
sudo -u postgres psql -d interjob_master -c "
SELECT relname, pg_size_pretty(pg_total_relation_size(c.oid))
FROM pg_class c JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE c.relkind = 'r' AND n.nspname = 'public'
ORDER BY pg_total_relation_size(c.oid) DESC LIMIT 10;"

# Index usage
sudo -u postgres psql -d interjob_master -c "
SELECT indexrelname, idx_scan, pg_size_pretty(pg_relation_size(indexrelid))
FROM pg_stat_user_indexes ORDER BY idx_scan DESC LIMIT 10;"

# Cache hit ratio
sudo -u postgres psql -c "
SELECT
  sum(heap_blks_hit) / nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0) as cache_ratio
FROM pg_statio_user_tables;"

## KEY LEARNINGS
================================================================================

1. ALWAYS use fcntl locks for cron scripts that may overlap
2. Parallel processing = 93x faster for large table operations
3. Materialized views eliminate slow GROUP BY queries
4. pg_stat_statements reveals actual bottlenecks
5. Kill competing queries before they cause deadlocks
6. Batch processing with state tracking = resumable operations
7. Trigram indexes (gin_trgm_ops) for LIKE queries
8. Consolidate small DBs into schemas for simpler management
9. Check for cross-batch duplicates after windowed dedup
10. Global pass needed after batch dedup for complete cleanup
